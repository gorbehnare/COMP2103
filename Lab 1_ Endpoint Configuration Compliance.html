Lab 1: Endpoint Configuration Compliance

OS Hardening using CIS Benchmarks

If you are designing a system, you can ensure security by having a full understanding of all the underlying technology in your environment and protocols. Acquiring all the required knowledge is certainly worth striving for. However, in reality, due to the vast amount of available information, the best most of us can do, is to gain general knowledge in some areas and specialize in a narrow field. Several organizations (such as CIS and OpenSCAP) are specialized in consulting with professionals and members with expertise in various fields and using consensus processes to create security configuration "benchmarks". These recommendations are available to individuals and organizations to assist in designing and configuring secure systems.

For this lab, we will be looking at the CIS benchmarks, and later on, evaluate our system using SCA profiles.

A Note on Ubuntu Desktop Version Selection

Canonical provides support for commercial customers by providing tooling for various frameworks and benchmarks. You can see the list here: https://ubuntu.com/security/compliance-automation

Most of the tools and scripts are provided under the Ubuntu Security Guide" (or USG). The USG scripts allow automation of auditing and compliance with various standards. Ubuntu Security Guide provides scripts that can automatically apply most CIS Benchmarks recommendations, at the expense of breaking non-default configurations. It is recommended to run the USG on a new installation for the best results, where you are not concerned with the existing configuration or data.

Visit the Ubuntu CIS Security page to find out which versions of Ubuntu have CIS tooling. In this lab, we will be targeting CIS controls for "Level 1 - Workstation".

Notice: As of this writing, Canonical only provides compliance and hardening profiles (USG) for Ubuntu up to Ubuntu 22.04 LTS. The Ubuntu 24.04 LTS and 26.04 LTS are not yet supported. Please make sure to choose the latest version of the automation tooling available for CIS.

Notice: If you desire to use the latest Ubuntu or any other software release without USG support or other CIS support, you can either pay for membership at CIS to get access to "build-kits" or build your own scripts using CIS benchmark documents. You are NOT allowed to share the scripts you build without a license from CIS. The CIS benchmarks are released under a custom "Creative Commons non-commercial use" and are non-permissive. You are not allowed to create or distribute a product or service using the benchmarks without being a paid member and obtaining a license from CIS. As a result, you may also have difficulty finding (free) automation tools from 3rd parties. This is a legal limitation, not a technical one.

Wazuh OS and Network Configuration Guide

  • Log in using credentials wazuh-user / wazuh
  • configure networking:
    • Use ip a show to view current interface names and IP addresses
    • Amazon Linux has network configuration scripts located at /etc/sysconfig/network-scripts
    • You can simply copy one interface configuration script to another file and just modify the first line to correspond to the correct interface name
      • for example: sudo cp ifcfg-eth0 ifcfg-eth1 && sudo nano ifcfg-eth1
        • Change the first line: DEVICE=eth1
      • Then run the script to apply the settings to the interface: sudo ifdown eth1 && sudo ifup eth1
      • Use ip a show to verify the settings are applied
  • Optional but highly recommended: If you are not doing the lab in one sitting, you can set the host-only interface to static IP so you will always get the same IP across reboots using the following example template (make sure to modify the IP addresses and interface names to match your own environment):
DEVICE=eth0       # ensures we have the correct interface name
BOOTPROTO=none    # dhcp is not required
ONBOOT=yes        # interface should be activated on boot
IPADDR=192.168.56.90
NETMASK=255.255.255.0
GATEWAY=0.0.0.0
BROADCAST=192.168.56.255
USERCTL=no
NM_CONTROLLED=no
DEVICE=eth0       # ensures we have the correct interface name
BOOTPROTO=none    # dhcp is not required
ONBOOT=yes        # interface should be activated on boot
IPADDR=192.168.56.90
NETMASK=255.255.255.0
GATEWAY=0.0.0.0
BROADCAST=192.168.56.255
USERCTL=no
NM_CONTROLLED=no
  • Once the NIC interface is configured and updated, you can run a software update using the YUM package manager: sudo yum update
  • Ensure you have access to your Wazuh web interface using a web browser pointed to the host-only IP address https://<ip-address>. The default username and password for the web interface is admin / admin

Deploying Wazuh Agent

Wazuh is an agent-based tool, meaning it will require an agent (software) to be installed on the endpoints that you would like to assess and monitor.

  • If you are interested, you can create a new Group for agents first. We only have one client system, so you may also use the default group.
  • In the Wazuh web interface, click on the navigation icon in the top left corner, and select Agent management. Click on the Deploy new agent button.
  • For your Ubuntu desktop client, select Linux and deb amd64 as the package.
  • Make sure the server address is set to the IP address of your Wazuh server. Assign the agent name to be <firstname>-<studentID>, for example: firstname-200512345.
  • Wazuh will generate CLI commands for downloading and installing the agent. If you have entered the correct information in the previous steps, you can just copy and paste the given command into your Ubuntu desktop terminal to get this package downloaded and installed.

Make sure to follow all steps to enable and start the agent service on your Ubuntu desktop. As soon as you start the service on your client system, you should see the agent added to Wazuh.

As you are working on the lab, you can force a re-evaluation of your client system by managing the Wazuh-agent service using sudo systemctl [stop | start | restart] wazuh-agent.

Conclusions and Final Thoughts

  • Looking at the list of failed and passed items in Wazuh, can you find out if having a 100% compliant system is possible with the default settings and adding any exceptions?
    • This should clearly demonstrate that security is porous, and complete security does not exist.
    • This also demonstrates that sometimes security comes at the cost of functionality and convenience.
      • Revisit section 4.1.7 Ensure ufw default deny firewall policy (Automated) where the default deny rule for outbound traffic will break functionality.
  • Looking at the failed items, you may find that some of them may not seem very "risky" in many situations.
    • Although there is a rationale behind security configuration recommendations, they are generic in nature and do not take specific situations and needs into consideration.
    • Risk is variable and is very specific to individual systems and situations.
  • If you are asked to create systems with complete and rigorous compliance requirements that not all systems can meet, a risk assessment report on non-compliant systems and failed items is probably the best way to communicate the reasons behind non-compliance to the authorizing authorities and management. This may allow you to get approval for "exceptions", although at the cost of some additional paperwork!
